## D-18 — Оператор/юрлицо: ИП на УСН (старт); возрастной порог — отложен
**Контекст:** шаг 2 launch-readiness — выбор юрлица/оператора ПДн и возрастной модели (`docs/STEP2_OPERATOR_AGE.md`).
**Решение:** оператор данных и продавец — **ИП на УСН** для старта (переход на ООО — отдельным решением при росте/партнёрах/инвестициях). Возрастной порог владелец **отложил** (обсудить позже/с юристом).
**Причина:** соло-MVP без партнёров — ИП быстрее/дешевле. Возраст требует доп. обдумывания: ограничение влияет на продвижение (теряем сегмент 14-17), но <18 без защиты = риск (геолокация несовершеннолетних, согласие родителей).
**Рекомендация Claude по возрасту (не принята/не отклонена):** 18+ на старте (мягкая самодекларация, минимум риска, рост почти не страдает) + подростковый режим 14-17 Фазой 2 (родительское согласие + приватность гео).
**Статус:** ИП — принято (2026-06-21). Возраст — OPEN (за владельцем). Дальше: юрист на тексты документов (шаг 3).

## D-17 — Production-инфраструктура и юридическая готовность до публичного запуска
**Контекст:** общий аккаунт, GPS, территории, покупки, баллы, клубы и будущие чаты
требуют внешней инфраструктуры и формального процесса запуска.
**Решение:** основной кандидат для production — **Yandex Cloud**; домен/DNS может
оставаться в Beget. Beget VPS/Cloud допустим для MVP, обычный виртуальный хостинг — нет.
Окончательный выбор — после сравнения стоимости, PostGIS, SLA, бэкапов, договора и
условий обработки данных. До публичного запуска обязателен `docs/LAUNCH_READINESS.md`:

Store/баллов/контента, HTTPS, бэкапы, мониторинг и аудит.
**Причина:** одна галочка при регистрации не закрывает юридические и технические риски;
архитектура должна поддерживать минимизацию данных и аудит согласий.
**Статус:** принято как направление и блокирующий launch-gate (2026-06-21);
провайдер ещё не закуплен.

## D-16 — Сглаживание GPS-трека: фильтр Калмана в приложении (не snap-to-road)
**Контекст:** сырой GPS «дрожит» (±3-10 м даже при хорошей точности) — линия маршрута и зелёная метка прыгают. Рассматривали: (A) сглаживание в приложении (Калман) + анимация метки; (B) привязка к дорогам (snap-to-road, OSRM/Mapbox); (C) гибрид (A в забеге, B в истории).
**Решение:** A — лёгкий фильтр Калмана (`features/run/data/gps_kalman.dart`) сглаживает позицию (вес фикса = его точность), линию и метку рисуем по сглаженной точке; метка анимированно «скользит» между фиксами. Сырой фикс по-прежнему отсекается по точности/скорости/прыжку до Калмана.
**Причина:** Квартал — игра про территории: захват территории идёт по *реальному* пути (двор/парк/срезка; полигон маршрута, НЕ гексы — см. D-09). Snap-to-road «притянул» бы трек к дороге → исказил захват и античит, плюс требует внешний сервис. Калман даёт и гладкую линию, и честный путь, офлайн и бесплатно. Snap-to-road — возможно позже только как «красивая картинка» готовой пробежки в истории.
**Статус:** A внедрён. Параметр процесса q≈3 м/с; при необходимости подстроить по полевым тестам.

## D-15 — Прод-конфиг через env/флаги сборки, dev-дефолты в коде
**Контекст:** все 3 поверхности на общем бэке; нужно уметь переключать API dev↔prod без правок кода и без коммита секретов (репо публичный). Хостинга пока нет.
**Решение:** dev-значения остаются дефолтами в коде (всё работает из коробки). Прод включается:
- backend — переменными окружения: `DJANGO_DEBUG`, `DJANGO_SECRET_KEY`, `DJANGO_ALLOWED_HOSTS`, `DJANGO_CORS_ORIGINS` (пуста → CORS открыт для dev; задана → только эти источники + CSRF);
- приложения — `--dart-define=SPORT_STORE_API_BASE_URL=…` / `KVARTAL_API_BASE_URL=…` (уже были; добавлен `KVARTAL_ZONES_URL`);
- сайт — `ecosystem.js` авто-определяет dev по `localhost`, иначе `PROD_API` (плейсхолдер-домен) либо `window.STAW_API_BASE`.
Чеклист — `docs/DEPLOY.md`, шаблон env — `backend/django_api/.env.example`.
**Причина:** один код для dev и прод, секреты только в окружении сервера, минимум ручной работы при деплое.
**Статус:** готово (PR прод-конфиг). Реальные домены/хостинг подставит владелец позже.

## D-14 — Модель территорий: гибрид (живой слой с распадом + вечный личный след)
**Контекст:** обсуждали, держать ли жёсткое 72ч-удержание или сделать вечную фиксацию (твоё/навсегда, перехват всегда).
Чистая «вечная» убивает ядро бегового приложения (нет повода бегать повторно, особенно пока мало игроков),
насыщает карту и растит геометрию без предела. Чистый распад 72ч — слишком наказывает за паузу (отток).
**Решение (выбор владельца):** ГИБРИД из двух слоёв:
- **Живой слой** (карта/рейтинг/клубы): территория держится **7 дней** (`HOLD_HOURS=168`) от последнего забега,
  перехват бегом — всегда; не подтвердил забегом → освобождается. Держит карту живой и мотивирует бегать.
- **Вечный личный след** (`footprints`): объединение всего, что когда-либо пробежал — растёт, НЕ уменьшается
  (ни временем, ни перехватом). Показывается в профиле как «исследовано N км²». Даёт чувство «моё навсегда»
  без насыщения соревновательной карты.
**Причина:** баланс мотивации бегать (выгодно нам) и удержания/эмоции владения; новичкам всегда есть место.
**Статус:** активно. Бэк: `territories` (живой, 7д) + `footprints` (вечный) + `GET /v1/footprint`. Клиент: карточка профиля.
Тонкая защита 24ч после захвата от мгновенного отбора — опционально, следующим шагом. Окна (7д/24ч) подкрутим по живому тесту.

## D-13 — Экосистема: единый бэк + контракт-first, подключаем поверхности постепенно
**Контекст:** развиваем 3 направления (Квартал, SportStore, Сайт STAW) как одну экосистему —
один аккаунт (SSO), общий баланс баллов (зарабатываются в Квартале, тратятся в Store и на сайте).
**Решение:** НЕ переписывать всё разом. Один бэкенд = источник правды (Django, готов). Контракт стабилен
(`ECOSYSTEM_API.md`). Каждый новый экран/функция: сначала эндпоинт в Django → затем подключаем поверхность
(приложение или сайт) по модулю, по очереди. Готовое не ломаем — расширяем. Баланс баллов — одно число на
сервере; любая поверхность зарабатывает (Квартал) или тратит (Store/сайт) его через один и тот же API.
**Порядок (по решению владельца):** (1) сначала ДОДЕЛАТЬ территории Квартала (D-09, на PostGIS),
(2) затем Store-магазин на бэке (каталог/заказы/списание баллов), (3) затем сайт (вход общим аккаунтом + баллы + витрина).
**Причина:** меньше риска и без простоя; так уже прошёл переезд на Django. Фундамент (1 аккаунт + общие баллы) уже живой.
**Статус:** принято (2026-06-15). Связано с D-06/D-12 (бэк), D-09 (территории).

## D-12 — Переезд бэка на Django + PostgreSQL/PostGIS в Docker (рядом с FastAPI)
**Контекст:** территории (D-09) требуют PostGIS; владелец выбрал полный переезд на Django.
**Решение:** бэкенд переводим на **Django + DRF + PostgreSQL/PostGIS**, запуск в **Docker** (`backend/docker-compose.yml`:
`db` = postgis/postgis, `web` = Django на хосте :8001). FastAPI (`backend/main.py`, :8000) **держим рабочим**
во время перехода (обслуживает приложения), переносим эндпоинты по модулям с сохранением JSON-контракта,
проверяем на устройстве, переключаем `baseUrl` приложений на Django и **только потом отключаем FastAPI**.
Цель — Django-only; FastAPI не удалять до паритета и проверки.
- **Почему Django в Docker:** не тащим `psycopg`/GDAL на хостовый Python 3.14 (нет колёс); всё в контейнере.
- **Пререквизит окружения:** Docker Desktop установлен, но нужен **WSL2** (`wsl --install`, админ+перезагрузка) и запуск Docker Desktop.
- **Порядок переноса:** accounts/auth → profile → loyalty → clubs → leaderboard → данные SQLite→Postgres → переключение приложений → территории (PostGIS, D-09).
**Причина:** PostGIS нужен для территорий; единый бэкенд на Django удобнее (админка, ORM, гео). Гладкий переход без простоя приложений.
**Статус:** ✅ ПЕРЕХОД ВЫПОЛНЕН (2026-06-15). Все эндпоинты на Django (auth/profile/loyalty/clubs/leaderboard),
данные SQLite→Postgres перенесены (скрипт `manage.py import_sqlite`), Django на :8000, приложения работают
против него (проверено на устройстве, токены совместимы), FastAPI остановлен (файл оставлен как legacy).
Дальше — территории на PostGIS (D-09); позже убрать legacy FastAPI и обновить CI на Django. Связано с D-06/D-07, D-09.

## D-11 — Рейтинг по КМ (не по баллам), периоды неделя/месяц
**Контекст:** по чему ранжировать рейтинг.
**Решение:** ранжируем по **пробежанным км**, НЕ по баллам — баллы тратятся в магазине, ранг бы прыгал (нестабильно/несправедливо).
- **км берём из ЗАРАБОТАННЫХ за бег записей**, а не из баланса кошелька: `км = Σ(loyalty_transactions.amount, source='runnerRun', за период) / 10`. Это значение только растёт (траты — отдельные записи `redeem`, в сумму не входят) → устойчиво к покупкам. Отдельного синка пробежек для рейтинга НЕ нужно — данные уже на бэке (начисления за бег, с датой).
- **Периоды:** `week` (Пн–Вс), `month`, `all`. Неделя — основная гонка, месяц — чемпион; в профиле — «всего км».
- **Эндпоинты:** `GET /v1/leaderboard/users?period=` и `GET /v1/leaderboard/clubs?period=` (ранг по км + своё место/клуб).
- **Районы** (вкладка 3): кто из клубов контролирует больше зон в районе города — нужен **захват территорий на сервере (D-09)**, пока заглушка.
- **Награды за неделю/месяц** (бейдж/бонус чемпиону) — этап 2 поверх базового рейтинга.
**Причина:** км — накопленное достижение, не уменьшается; баллы — валюта, для рейтинга не годится (идея владельца).
**Статус:** backend готов и проверен (2026-06-15). UI — следом. Связано с D-09 (районы), D-02.

## D-10 — Клубы: бэкенд-фича, один клуб на человека, политику вступления выбирает владелец
**Контекст:** Клуб — многопользовательская функция; реальные данные требуют бэкенда.
**Решение:** клубы на общем бэкенде (FastAPI сейчас → Django позже, контракт сохранить). Правила:
- один пользователь = максимум один клуб (enforced уникальным индексом `club_members.user_id`);
- создать клуб может любой авторизованный (становится владельцем/owner);
- владелец при создании выбирает политику вступления: `open` (сразу) или `request` (по заявке с подтверждением);
- роли: `owner` / `member` (admin — позже);
- агрегаты клуба считаются по БАЛЛАМ участников (км пока нет на бэке — нужен run-sync, отдельная задача);
- логотип — пресет (emoji/строка) на старте; загрузка картинок — позже (нужно хранилище/CDN).
**Эндпоинты:** `GET/POST /v1/clubs`, `GET /v1/clubs/me`, `GET/PATCH /v1/clubs/{id}`,
`POST /v1/clubs/{id}/join|leave`, `GET /v1/clubs/{id}/requests`, `POST /v1/clubs/requests/{id}/approve|reject`.
**Причина:** один клуб/человек — чтобы рейтинг клубов имел смысл; политика вступления per-club — по решению владельца (запрос пользователя).
**Статус:** backend готов и проверен (2026-06-14), frontend (вкладка Клуб) — следом. Связано с D-06 (Django), D-02 (сервер — источник правды).

## D-09 — Модель территорий «Квартал»: по реальному маршруту, НЕ гексы
**Контекст:** как захватывать и отображать территории и при этом тянуть 1000–3000 игроков без тормозов.
**Решение:** территория = площадь ВНУТРИ реального бегового маршрута игрока (произвольный полигон),
тематически — настоящие районы/кварталы. **H3-гексы ОТКЛОНЕНЫ** (ломают суть и вид «Квартала»).
Заранее заготовленные блоки из `zones.json` как единица владения тоже уходят.
- **Отрисовка:** полупрозрачная заливка цветом владельца + насыщенный контур; карта (дома/улицы) видна
  насквозь. Цвета: свои / клуб / чужие / свободно. Точный оттенок и прозрачность — дизайн, пока не утверждаем.
- **Захват:** замкнул петлю → область внутри = территория. Возвращаться в точку старта НЕ обязательно:
  маршрут, начавшийся на своей территории и снова вошедший в неё, замыкается о собственную границу
  (касание в 2 точках A/B), новый кусок прирастает (`ST_Union`). Полная отдельная петля вне своей
  территории = новая отдельная территория.
- **Перехват чужого:** пересечение с чужой территорией отрезается у врага и переходит игроку (`ST_Difference`).
- **Чистка/сглаживание при фиксации (сервер, один раз):** выкинуть мусорные точки и GPS-прыжки →
  упростить (Дуглас–Пекер) → скруглить углы (Chaikin/сплайн) → проверить геометрию (без самопересечений).
  Форма остаётся реальным маршрутом; сила сглаживания тюнится.
- **Производительность:** хранить точный контур, на отдалённом зуме отдавать упрощённо; грузить только
  видимую область (bbox + zoom); кэш; реалтайм-дельты — позже.
- **Источник правды — сервер:** владение, пересечения, срок удержания 72 ч, античит (скорость ≤ 40 км/ч,
  адекватность GPS). Клиент рисует оптимистично, затем сверяется с сервером.
- **Геометрия — PostGIS** (`ST_Union` / `ST_Difference` / `ST_Simplify*` / `ST_MakeValid`).
- **Текущее расхождение (переделать):** карта берёт зоны со старого Go-бэка `localhost:3000` и красит
  заготовленные блоки; владение лежит локально (SharedPreferences). Зерно нужной модели уже есть —
  `CapturedArea` хранит полигон реального маршрута.
**Причина:** «Квартал» = захват реальных районов по своему бегу; гексы ломают суть и красоту.
Нужна плавность при 1000–3000 игроках без багов.
**Статус:** принято (2026-06-14), реализация поэтапно вместе с переходом на Django/PostGIS (D-06).
Связано с D-02 (сервер — источник правды), D-06 (Django/PostGIS).

## D-02 — SSO по телефону, единый аккаунт и общий баланс
**Решение:** один `user_id` на телефон; вход по телефону (dev-код `1234`) в Квартале и Store ведёт в один аккаунт;
баллы — общий счёт (`/loyalty/*`). Бек — источник правды, локальное хранилище приложений — кэш.
**Причина:** ядро смысла экосистемы — единый пользователь и баллы, заработанные в Квартале, тратятся в Store.
**Статус:** активно, доказано на устройстве.

## D-01 — Дизайн/бренд пока не трогаем
**Решение:** не менять цвета/шрифты/брендинг/вёрстку. Только структура, корректность, обмен данными.
**Причина:** дизайн не утверждён, может кардинально поменяться. Сначала — чтобы всё работало без багов.
**Статус:** активно (директива владельца).
