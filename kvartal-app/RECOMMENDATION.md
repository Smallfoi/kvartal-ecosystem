## ЧАСТЬ 1: ОБЩАЯ СТРАТЕГИЯ — ЧЕМУ УЧАТ ТОПЫ

### Главный урок от Nike и Adidas

Топовые бренды делают **не магазин**, а **экосистему**. Разница принципиальная:

- Магазин = пришёл, купил, ушёл
- Экосистема = пришёл, вернулся, тренировался, получил баллы, купил снова, порекомендовал другу

На старте нам не нужна экосистема как у Nike — нам нужно **заложить фундамент**, чтобы к ней прийти. Этот документ описывает именно такой фундамент.

## ЧАСТЬ 2: АРХИТЕКТУРА УПРАВЛЕНИЯ ПРИЛОЖЕНИЕМ

### 2.1 Что должно быть в Admin Panel

Приложение без админ-панели — это машина без руля. Все топ-бренды управляют контентом через внутренние CMS-инструменты. Для нашего проекта нужна веб-панель с **пятью модулями**:

```
ADMIN PANEL (веб-интерфейс для менеджеров)
│
├── 1. ТОВАРЫ (Product Manager)
│       ├── Добавление / редактирование / архивация товара
│       ├── Загрузка медиа (фото + видео)
│       ├── Управление размерами и цветами
│       ├── Остатки на складе (или синхронизация с 1С)
│       └── Теги: новинка, хит, рекомендуем, скидка
│
├── 2. КОНТЕНТ (Content Manager)
│       ├── Баннеры главного экрана
│       ├── Порядок категорий
│       └── Акционные страницы / лендинги
│
├── 3. АКЦИИ И СКИДКИ (Promotions Engine)
│       ├── Скидки на товар / категорию / бренд
│       ├── Flash Sale с таймером
│       ├── Промокоды
│       └── Подарки при покупке от суммы X
│
├── 4. ЗАКАЗЫ (Order Manager)
│       ├── Список заказов + фильтрация по статусу
│       ├── Изменение статуса заказа
│       └── Уведомление клиента при изменении статуса
│
└── 5. ПОЛЬЗОВАТЕЛИ (User Manager)
        ├── Список клиентов
        ├── История покупок
        └── Push-рассылки по сегментам
```

### 2.2 Рекомендуемый технический стек бэкенда

Учитывая текущий план проекта (Node.js/FastAPI + PostgreSQL + 1С + ЮКасса):

```
Рекомендую: FastAPI (Python) + PostgreSQL + Redis

Почему FastAPI, а не Node.js:
- Автоматическая документация API (Swagger/OpenAPI) — удобно для команды
- Быстрее интегрировать с ML (если добавите рекомендации)
- Проще синхронизация с 1С через python-библиотеки

Хранилище медиа: Yandex Object Storage (уже запланировано — верное решение)
Кэш: Redis — для баннеров, каталога, популярных товаров
Очередь задач: Celery — для отправки push-уведомлений, email
Push-уведомления: Firebase FCM (уже запланировано)
```

## ЧАСТЬ 3: УПРАВЛЕНИЕ ТОВАРАМИ — КАК ЭТО ДЕЛАЮТ NIKE И ADIDAS

### 3.1 Структура карточки товара

Nike использует минимум **8–12 фотографий** на товар. Adidas добавляет **lifestyle-видео 15–30 секунд**. Gymshark использует TikTok-стиль вертикальное видео прямо в карточке.

**Для нашего проекта** модель `Product` уже хорошая база. Рекомендую расширить её:

```
Текущая модель (уже есть):          Что добавить:
─────────────────────────────────────────────────────
id, name, brand, categoryId         + subcategoryId (для фильтрации)
price, oldPrice                     + (уже есть — хорошо)
imageUrls (список)                  + videoUrl (одно видео на товар)
description                         + shortDescription (для карточки в списке)
sizes, colors                       + stockPerSizeColor (остатки по вариантам)
isNew, isFeatured                   + isOnSale, isBestseller (теги)
rating, reviewCount                 + (уже есть — хорошо)
inStock                             + stockCount (точное кол-во)
                                    + materialComposition (состав ткани)
                                    + careInstructions (уход)
                                    + weight (для расчёта доставки)
```

### 3.2 Медиа-стратегия (фото и видео)

**Урок от Gymshark:** они продают лосины за $60 через видео в карточке. Статичные фото конвертируют хуже.

**Рекомендую для каждого товара:**

| Тип медиа | Количество | Формат | Назначение |
|---|---|---|---|
| Фото на белом фоне | 1 | 800×800px | Миниатюра в каталоге |
| Фото 360° | 3–4 | 1200×1200px | Галерея в карточке |
| Lifestyle-фото | 2–3 | 1200×1600px | "В действии" |
| Видео | 1 | MP4, 15–30 сек, 9:16 | Автозапуск в карточке |

**Техническая реализация:**
- Загрузка в Yandex Object Storage
- Автоматическое сжатие через backend при загрузке (библиотека `Pillow` для Python)
- CDN для быстрой отдачи (Yandex CDN или встроенный в Object Storage)
- В приложении уже есть `cached_network_image` — это верное решение

### 3.3 Синхронизация с 1С

У вас уже есть 1С-разработчик — это ключевое преимущество. Рекомендуемая схема:

```
1С (источник правды)  →  REST API 1С  →  Backend (FastAPI)  →  PostgreSQL  →  Приложение
     ↑                                         ↓
  Остатки,                              Доп. данные:
  цены,                                 фото, видео,
  наличие                               описания, теги
```

**Важно:** 1С должна быть источником только для цен и остатков. Описания, фото, теги, видео — управляются через Admin Panel. Не доверяйте описания 1С — они всегда технические и не продают.

## ЧАСТЬ 4: УПРАВЛЕНИЕ АКЦИЯМИ И СКИДКАМИ

### 4.1 Что используют топ-бренды

**Nike:** персональные скидки через Members-Only, geo-push уведомления  
**Adidas:** adiClub баллы + Flash Sale с таймерами + CONFIRMED для хайп-релизов  
**Gymshark:** "Drops" — ограниченные по времени и количеству релизы, создающие ажиотаж

### 4.2 Система акций для нашего проекта

В приложении уже есть поле `oldPrice` в модели Product — это база для скидок. Нужно расширить:

**Таблица Promotions в БД:**
```sql
promotions
  id, name, type (percentage | fixed | free_shipping | gift)
  value (сумма или процент скидки)
  condition_type (order_amount | category | product | brand | first_order)
  condition_value (например: categoryId или минимальная сумма)
  promo_code (если промокод, иначе null)
  starts_at, ends_at
  is_active, usage_limit, usage_count
  show_timer (показывать таймер обратного отсчёта)
```

**Приоритеты акций** (рекомендую следующий порядок):
1. Промокод (наивысший приоритет — клиент ввёл сам)
2. Flash Sale (временная акция)
3. Скидка на категорию
4. Скидка на бренд
5. Скидка на товар (oldPrice в карточке)

### 4.3 Flash Sale — обязательный инструмент

В приложении есть `flutter_animate` — идеально для таймера обратного отсчёта.

**Рекомендую добавить в HomeScreen:**
- Блок "Акция дня" с таймером (уже есть баннеры — можно расширить)
- Таймер показывает часы:минуты:секунды до конца акции
- Push-уведомление за 2 часа до конца Flash Sale

**Урок от Adidas:** flash sale увеличивает конверсию на 40–60% в период акции. Создаёт срочность.

## ЧАСТЬ 5: ПРОГРАММА ЛОЯЛЬНОСТИ — ИНСАЙТ ОТ ADIDAS adiClub

### 5.1 Почему лояльность важна с первого дня

adiClub у Adidas — лучшая программа лояльности в нише. Её секрет: **баллы начисляются не только за покупки**.

**Рекомендуемая модель для нашего проекта:**

| Действие | Баллов |
|---|---|
| Покупка | 1 балл за каждые 10 руб. |
| Первая покупка | +50 бонусных баллов |
| Отзыв на товар (с фото) | +10 баллов |
| Регистрация | +20 баллов |
| День рождения (месяц) | +30 баллов |
| Приглашение друга | +50 баллов |

**Уровни:**

| Уровень | Баллов | Привилегии |
|---|---|---|
| Базовый | 0–199 | Базовый кэшбэк 1% |
| Серебро | 200–499 | Кэшбэк 2% + ранний доступ к акциям |
| Золото | 500–999 | Кэшбэк 3% + бесплатная доставка |
| Платина | 1000+ | Кэшбэк 5% + эксклюзивные товары + VIP-поддержка |

### 5.2 Техническая интеграция

Модель `AuthUser` уже есть. Добавить:
- `loyaltyPoints: int` к модели пользователя
- `loyaltyLevel: LoyaltyLevel enum` (базовый/серебро/золото/платина)
- Таблица `loyalty_transactions` в БД (история начислений/списаний)
- Экран "Мои баллы" в Profile (рядом с вкладками Orders/Wishlist)

## ЧАСТЬ 6: PUSH-УВЕДОМЛЕНИЯ — КАК ПРАВИЛЬНО

### 6.1 Что делают топ-бренды

Nike использует **geo-push** (уведомление когда рядом с магазином). Adidas — **таргетированные push** по интересам. Gymshark — **countdown push** ("Акция заканчивается через 2 часа").

### 6.2 Правила для нашего проекта

В приложении уже есть NotificationsProvider и экран уведомлений. Firebase FCM запланирован. Рекомендую **6 типов уведомлений**:

| Тип | Когда | Пример |
|---|---|---|
| Статус заказа | При каждой смене статуса | "Ваш заказ #1234 передан в доставку" |
| Flash Sale | За 2 часа до конца | "Скидки до 50% заканчиваются через 2 часа!" |
| Новинки | Еженедельно (по интересам) | "Новые кроссовки Nike — смотрите первыми" |
| Брошенная корзина | Через 2 часа после ухода | "В вашей корзине ждут товары" |
| Персональная скидка | 1 раз в месяц | "Только для вас: -15% на ваш любимый бренд" |
| Акция дня | В 10:00 в день акции | "Сегодня -30% на все куртки" |

**Важное правило (урок от всех брендов):** не более **2 push в неделю** на пользователя. Частые уведомления — главная причина удаления приложения.

## ИТОГО: ТРИ ГЛАВНЫХ ИНСАЙТА

**1. Медиа решает**
Gymshark обогнал компании с 20-летней историей потому что делал лучший видео-контент в карточках товаров. Один 15-секундный ролик продаёт лучше 10 фотографий.

**2. Лояльность = ежедневный контакт**
Nike и Adidas создали фитнес-приложения не ради здоровья пользователей. Это канал ежедневного контакта с брендом. Наш путь на старте — программа баллов, которая даёт причину возвращаться.

**3. Простота выигрывает у перегрузки**
Nike — самый дорогой бренд с самым простым приложением. Не добавлять фичи ради фич. Каждый экран должен решать одну задачу.

## ЧАСТЬ 11: ЭКОСИСТЕМА — ТРИ ПРИЛОЖЕНИЯ, ОДНО ЯДРО

### 11.1 Видение

Проект развивается не как одно приложение, а как **экосистема из трёх взаимосвязанных продуктов** — точно по модели Nike (NRC + Nike App) и Adidas (Running + Shop + adiClub):

| Продукт | Платформа | Назначение |
|---|---|---|
| **Runner App** | Flutter (iOS + Android) | Бег, соревнования, захват территорий, геймификация |
| **Sport Store App** | Flutter (iOS + Android) | Каталог товаров, покупки, баллы как скидки |
| **Сайт** | Web | То же что Store App, но в браузере |

Все три продукта — разные "окна" в **одну общую базу данных и один бэкенд**.

### 11.3 Поток баллов — как работает между приложениями

```
Пользователь пробежал 5 км в Runner App
              ↓
Бэкенд начисляет баллы на единый аккаунт
              ↓
Открывает Sport Store App или сайт
              ↓
В корзине видит: "У вас 430 баллов = скидка 430 ₽"
              ↓
Применяет баллы → оформляет заказ
              ↓
Баллы списаны. История обновлена во всех трёх продуктах
```

### 11.4 Правила начисления баллов

**В Runner App:**

| Действие | Баллов |
|---|---|
| Пробег 1 км | +10 баллов |
| Захват территории (первый раз) | +50 баллов |
| Удержание территории (каждый день) | +15 баллов |
| Победа в соревновании | +200 баллов |
| Личный рекорд на дистанции | +30 баллов |
| Streak 7 дней подряд | +100 баллов |

**В Sport Store / Сайте:**

| Действие | Баллов |
|---|---|
| Покупка | +1 балл за каждые 10 ₽ |
| Первая покупка | +50 бонусных баллов |
| Отзыв с фото | +10 баллов |
| Регистрация | +20 баллов |

### 11.5 Правила использования баллов

| Параметр | Значение |
|---|---|
| Курс | 1 балл = 1 ₽ скидки |
| Максимум списания | 30% от суммы заказа |
| Срок действия баллов | 12 месяцев с момента начисления |
| Минимальный остаток для применения | 50 баллов |

### 11.7 Единый аккаунт (SSO)

Один логин/пароль работает во всех трёх продуктах. JWT-токен единый.

```
Зарегистрировался в Runner App
→ Тот же аккаунт сразу работает в Sport Store и на сайте
→ Баллы, история покупок, настройки — всё синхронизировано
```

**Важно:** в текущем Sport Store App авторизация уже построена на `AuthProvider` с поддержкой JWT (`ApiAuthRepository`). При создании Runner App использовать те же эндпоинты `/auth/login` и `/auth/register` — не создавать отдельную авторизацию.

### 11.9 Почему это работает — урок от Nike и Adidas

Nike создал Nike Run Club не потому что хотел помочь людям бегать. NRC — это **канал ежедневного контакта** с брендом. Пользователь открывает приложение каждый день → видит Nike → покупает Nike.

У Adidas та же история с Adidas Running (бывший Runtastic).

**Наша экосистема делает то же самое, но лучше:** Runner App не просто трекает бег — он создаёт соревновательную механику вокруг территорий. Это значит пользователь возвращается не ради статистики, а ради победы над соседом. А баллы превращают каждый километр в скидку на кроссовки — прямая связь между активностью и покупкой.

### 11.10 Уникальное конкурентное преимущество

Nike и Adidas дают баллы только за покупки + тренировки. Механика **захвата территорий** — это то, чего нет ни у одного спортивного бренда. Это геймификация городского пространства, которая:

- Создаёт ежедневную причину открыть приложение
- Генерирует соревнование между пользователями (виральность)
- Привязывает к локальному сообществу (сложнее бросить)
- Конвертирует физическую активность в покупательную способность
